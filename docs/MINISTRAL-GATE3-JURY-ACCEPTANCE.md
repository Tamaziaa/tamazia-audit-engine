## Unit 1: the OpenRouter/Ministral provider (llm/router.js)
- Add an OpenRouter provider: endpoint `https://openrouter.ai/api/v1/chat/completions`, model `mistralai/ministral-8b`, auth `Bearer $OPENROUTER_API_KEY` read from env at call time only (never committed, never logged; Rule 16). Provider family label `mistral`, genuinely distinct from `llama`/`groq`/`gemini`/`qwen`/`nemotron` (C-133).
- Wrapped in the router's existing hard-deadline primitive (Rule 9), structured/JSON response mode set (C-137), one attempt then fail (no retry storm, C-138). A missing key marks the provider ABSENT loudly and the caller falls back or abstains, never silently degrades (C-135).
- The live network call is dependency-injected (a `fetchImpl`/provider factory) so every test is network-free; no real OpenRouter call runs in CI (the deadline-audit and no-network doctrine).

## Unit 2: Gate-3 entailment routed to Ministral (llm/entailment.js seam)
- The entailment (`checkEntailment`) call routes to Ministral-8b as the PRIMARY provider. On provider-absent/deadline/error it falls back to the existing free chain; if all fail, it returns an abstaining result so the finding demotes to needs_review (fail-closed, never a fabricated label).
- The entailment behaviour, gates and prompt are otherwise UNCHANGED (the glossary bridge, the atomic-claim hypothesis, the closed 3-label enum, the sanitisation door all stay). Only the provider preference changes.
- Benchmark A-B: a network-free test proves the entailment call attempts Ministral first (via the injected provider), and on Ministral failure falls to the free chain, and on total failure abstains. The existing llm-evals stay precision 1.000.

## Unit 3: wire Gate 5 (the diverse jury) into the adjudicator for shipping violations
- Today the adjudicator ships a `violation` on a single first-success model call. For any candidate that would ship as `violation` (the highest-stakes output), require a diverse-jury quorum before it ships: at least 3 legs across genuinely distinct provider families, ANCHORED by Ministral-8b (the reliable model) plus two other distinct families available from the estate chain (e.g. Groq/Llama + Gemini, or + DASHSCOPE/Qwen). Any single leg voting no_breach/insufficient VETOES the violation, which then demotes to needs_review (veto-to-reject, fail-closed).
- IMMUNITY (Rule 12 gate 5): the jury may never veto a curated catalogue fact (a register-verified fact, or a SECTOR_CORE/SECTOR_AGNOSTIC immune item); those bypass the jury. The jury judges only the model-adjudicated text breach.
- If fewer than the required distinct families are configurable (keys absent), the candidate cannot reach the jury quorum and demotes to needs_review (fail-closed; a violation never ships un-juried). Log the jury composition and each leg's vote (Rule 11).
- Benchmarks: (a) a config test asserts the jury has >= 3 distinct provider families (C-133); (b) a network-free test proves a candidate that all legs affirm ships `violation`, a candidate any leg vetoes demotes to `needs_review`, and a curated/immune fact bypasses the jury and is never vetoed; (c) a keys-absent test proves the violation demotes to needs_review rather than shipping un-juried.
